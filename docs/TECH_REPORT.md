# Техотчет: URL Shortener API

## Зачем делал
Второй проект хотел сделать более "системным": кэш, ограничения по запросам, TTL и аккуратная обработка edge-cases. Идея была показать, что умею думать не только про happy path.

## Что реализовал
- `POST /links` - создать короткую ссылку (можно задать alias).
- `GET /{code}` - редирект на оригинальный URL.
- `GET /links/{code}/stats` - статистика (клики).
- TTL для ссылок: после срока жизни возвращается `410`.
- Rate limiting по IP.
- Redis-кэш для резолва коротких ссылок.

## Как устроен проект
- `app/api/links.py` - endpoint-ы.
- `app/services/shortener.py` - создание/резолв ссылок.
- `app/services/cache.py` - подключение Redis.
- `app/services/rate_limit.py` - ограничение запросов.
- `app/models/models.py` - таблица `links`.
- `alembic` - миграции.

## Ключевые решения
- **Кэш с fallback.** Если Redis недоступен, API не падает, а идет в БД. Для pet-проекта это полезный компромисс между надежностью и простотой.
- **Rate limiting отдельно от роутов.** Логику проще переиспользовать и тестировать.
- **TTL на уровне бизнес-логики.** При резолве ссылки сразу проверяю срок и возвращаю корректный статус.

## Что было неидеально и как поправил
- На старте тесты падали из-за импорта `redis` в окружении запуска. Доустановил зависимость и проверил через `py -m`.
- Линтер поймал стиль в миграциях/импортах, прогнал `ruff --fix` и оставил чистый результат.
- По мере реализации упростил формулировки README и отчет, чтобы не выглядело "канцелярно".

## Тестирование
Проверяю три основных сценария:
- создание ссылки;
- редирект + инкремент счетчика;
- коллизия alias.

Команды:
- `py -m ruff check .`
- `py -m pytest`

Текущее состояние: тесты проходят (`3 passed`).
